How did I do that?

So, in order to call the map you have to use javascript.

A series of things are happening in this example. First you create a variable called mapwrapper. With that var we, on the 'document' create a 'div' called mapwrapper. We also subsequently create a css class called mapwrapper so that we can use the stylesheet to adjust the layout and look of the div.

Next, we do the exact same thing that we did with mapwrapper, but this time we're creating a div called mapcanvas. Now map canvas is the actual functioning map that calls the Google code and generates the map. Map wrapper is a container that we plan to put around mapcanvas in order to make it responsive to our CSS! So far so good? Good. Next we call, in the 'document', and query to find a selector called 'map'. This is the first time we're actually assigning anything to an existing html element. If you inspect element you'll see the article with a class of 'map'. So what happens with appendChild is that we are inserting the div of mapwrapper we created into the last child spot of the article. We run this again but instead insert the id mapcanvas inside the mapwrapper class. OMG it's so simple!

Super Useful links for Front End Web Development!

      HTML/CSS

    • A practical guide to HTML and CSS is written by Shay Howe. It's a great starting point for anyone who wants to know where they stand on their coding skillz.
    • Animate.css is a great page that shows you the code for the animations that you want using only CSS.
    • CanIuse.com is a fantastic resource that uses JSON files that are constantly updated in order to show you which browsers and their versions support your HTML/CSS features.
    • CSS-Tricks is a really useful and constantly updated page about CSS...tricks...
    • CSS Wizardry Harry Roberts is a big player in the front-end world. Look what he has to say. It's probably important.
    • Document Object Model or the DOM, what's that? This is Chris Coyier's nice explanation of what the DOM actually is.
    • HTML5 Cheat Sheet list of html5 tags. Could be very useful for writing semantically correct HTML.
    • HTML5 Outliner is a tool that you paste your HTML into and it breaks it down into a reader generated outline. This is a great tool for judging how semantically correct your HTML actually is!
    • Resizer is an awesome tool created by Malte Wassermann to show adjustable screen sizes. This lends its self to responsive web design nicely.
    • Responsive Tables a css-tricks article on creating responsive tables that look great on mobile and comp.
    • Scalable and Modular Architecture for CSS is an approach to web development written by Jonathan Snook. It's a quick read that's worth it.
    • Screen Sizes a list of screen sizes for every relevant device out there!
    • CSS Lint is an awesome tool for cleaning up your CSS. Redundancy is the enemy here. We're learning to completely utilize our code for efficiency and readability for both human and machine.
    • Sass Meister is a project that our fearless leader Dale Sande is a major part of. Very valuable tool if your using Sass to write CSS.

      Javascript/jQuery

    • jQuery is a library of callable Javascript that can be implemented to make everything on your website more responsive and awesome.
    • jQuery Cheat Sheet is Oscar Otero's awesome well organized cheat sheet of your most useful jQuery calls.
    • Code Pen is a great tool for writing and testing code. It's very similar to JsBin

Blog post #3: 4/8/2014

Semantic html!

Before we dive head deep into writing code, Dale has been impressing upon us the importance of semantics when writing code; focusing specifically on HTML5. By semantics I mean literally what the word actually means. There's the old saying when two people are arguing in agreement that "we're just arguing semantics" but the dev world is trying its best to establish a world wide standard of just what and how you're supposed to tag class or call things in HTML. Nicolas Gallagher writes an interesting article about it.

We're also implementing semantics for the purpose of UX (user experience). In the wild west that was the first days of writing html and css semantics too a back seat to the back end code and "as long as the page works" dev's used to be happy. Dale and a select few, including me and my 13 other classmates, are trying to change that not just for better code but to make the Internet universally accessible throughout the world.

Current access to the Internet for people with disabilities is atrocious and we're going to learn to change that!
